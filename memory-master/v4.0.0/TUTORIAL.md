## 快速回顾

### 什么是 Memory-Master？

Memory-Master 是一个**AI 记忆系统**，解决"AI 失忆"问题。

**核心痛点**：
> 每次打开一个新的 AI 会话，我都要重新告诉它：我是谁，我们在做什么，上次讨论到了哪里。

**解决方案**：
- ✅ 自动捕捉重要信息
- ✅ 持久化存储记忆
- ✅ 智能检索和召回
- ✅ 无需重新介绍

### 5 分钟安装

```bash
# 1. 克隆仓库
git clone https://github.com/your-username/memory-master.git
cd memory-master

# 2. 安装依赖
npm install

# 3. 配置 API Key
cp .env.example .env
# 编辑 .env，填入你的 API Key

# 4. 运行测试
npm test
```

看到 ✅ 表示安装成功！

## 核心概念

### 四层记忆模型

| 类型 | 说明 | 示例 | 存储位置 |
|------|------|------|---------|
| **情景记忆** | 具体事件、对话记录 | "2026-04-07 讨论了 Memory-Master v4.0" | memory/YYYY-MM-DD.md |
| **语义记忆** | 提炼的知识、概念 | "Memory-Master 基于 Karpathy 方法论" | MEMORY.md#语义 |
| **程序记忆** | 操作技能、流程 | "如何安装 Memory-Master" | MEMORY.md#程序 |
| **人设记忆** | 用户偏好、习惯 | "用户名叫 Jake，喜欢吃川菜" | MEMORY.md#人设 |

### 记忆生命周期

```
┌─────────────┐
│  1. 捕捉    │ ← 自动监听对话
└──────┬──────┘
       ↓
┌─────────────┐
│  2. 过滤    │ ← 敏感数据检测（16 种）
└──────┬──────┘
       ↓
┌─────────────┐
│  3. 压缩    │ ← 3 阶段压缩（原始→摘要→关键事实）
└──────┬──────┘
       ↓
┌─────────────┐
│  4. 存储    │ ← 按时间树组织
└──────┬──────┘
       ↓
┌─────────────┐
│  5. 检索    │ ← 5 种查询类型
└──────┬──────┘
       ↓
┌─────────────┐
│  6. 回写    │ ← 查询结果成为新记忆（累积效应）
└─────────────┘
```

### Karpathy 五环节

```
1️⃣ 原始数据输入   → 对话记录 + 元数据
2️⃣ 数据摄取与编译 → AI 提炼成结构化知识
3️⃣ 前端信息查看   → 时间树/主题/知识图谱
4️⃣ 复杂 QA 沉淀   → 查询结果回写
5️⃣ 工具校验增强   → AI 自动维护质量
```

## 高级功能

### 知识图谱（v4.0 新特性）

**什么是知识图谱？**

知识图谱是记忆之间的**交叉引用网络**，让知识产生"累积效应"。

**示例**：
```
Jake（人设）
  ├── 喜欢 → 川菜（偏好）
  ├── 参与 → Memory-Master 项目（项目）
  └── 创建 → 2026-04-07 讨论（事件）

Memory-Master 项目
  ├── 基于 → Karpathy 方法论（知识）
  ├── 创建者 → Jake（人设）
  └── 版本 → v4.0（状态）
```

**优势**：
- ✅ 支持复杂查询（"Jake 的项目基于什么方法论？"）
- ✅ AI 自动发现新关联
- ✅ 无需人工维护

---

### 复杂 QA 沉淀

**什么是复杂 QA？**

复杂 QA 不是简单检索，而是**综合多个记忆**生成深度回答。

**示例**：

```
你：Memory-Master 项目为什么选择 Karpathy 方法论？

AI: 🤔 综合分析：

1. **传统 RAG 的局限性**（2026-04-07 讨论）
   - 知识没有"累积效应"
   - 每次提问都要重新翻阅文档

2. **Karpathy 的核心解法**（2026-04-07 讨论）
   - AI 驱动的维基百科
   - 新文档加入时自动编织进知识网

3. **项目目标**（MEMORY.md#语义）
   - 解决 AI 失忆问题
   - 让记忆会"生长"

**结论**：选择 Karpathy 方法论是因为它解决了传统 RAG 没有累积效应的问题，
符合 Memory-Master 让记忆会"生长"的目标。
```

**实现原理**：
1. 查索引 → 定位相关记忆
2. 读摘要 → 获取概要信息
3. 找关联 → 遍历知识图谱
4. 回看原始证据 → 确认准确性
5. 生成报告 → 综合回答
6. **结果回写** → 成为新记忆

### 自动校验增强（Dreaming）

**什么是 Dreaming？**

Dreaming 是**后台自动巡检机制**，AI 定期检查和优化记忆质量。

**巡检内容**：

```
├── 检查重复记忆 → 合并
├── 检查缺失信息 → 标记
├── 检查链接断裂 → 修复
├── 检查术语不一致 → 统一
└── 发现新的概念关系 → 建立关联
```

**执行时机**：
- 低频任务（每天 1-2 次）
- 系统空闲时自动执行
- 不干扰正常使用

**示例输出**：
```
🌙 Dreaming 巡检报告（2026-04-07 凌晨）

✅ 检查通过：
- 无重复记忆
- 无缺失信息
- 无链接断裂

✨ 新发现：
- "Jake" 和 "用户" 是同一人 → 已建立关联
- "Memory-Master" 和 "Karpathy" 强相关 → 已增加权重

📊 统计：
- 总记忆数：127
- 知识图谱节点：45
- 交叉引用：89
```

## 最佳实践

### 1. 如何写好记忆提示

**❌ 错误示例**：
```
记住这个很重要！！！
```

**✅ 正确示例**：
```
记住：我每天早上 7 点起床，这是人设记忆。
```

**原则**：
- 明确记忆类型（人设/情景/语义/程序）
- 提供足够上下文
- 简洁明了

---

### 2. 如何查询记忆

**❌ 模糊查询**：
```
那个什么...你记得吗？
```

**✅ 精确查询**：
```
昨天讨论的 Memory-Master 项目基于什么方法论？
```

**查询技巧**：
- 使用具体时间（"昨天"、"上周"）
- 使用具体主题（"Memory-Master 项目"）
- 使用查询类型提示（"如何做 XXX" = 流程查询）

---

### 3. 如何管理记忆

**定期整理**（可选）：
```bash
# 查看记忆统计
npm run memory-stats

# 手动压缩
npm run compact

# 导出备份
npm run export
```

**记忆清理**：
- 情景记忆：自动按时间清理（7 天/30 天）
- 人设记忆：永久保留，手动删除
- 语义记忆：定期合并相似内容

---

### 4. 如何保护隐私

**敏感数据自动过滤**：
```
❌ "我的密码是 123456" → 过滤，不存储
❌ "我的信用卡号是 XXXX" → 过滤，不存储
✅ "我喜欢吃川菜" → 正常存储
```

**手动删除敏感记忆**：
```bash
# 查看敏感记忆
npm run view-sensitive

# 删除指定记忆
npm run delete-memory <id>
```

## 故障排查

### 问题 1: 记忆没有保存

**可能原因**：
- API Key 配置错误
- 网络问题
- 记忆过滤误判

**解决方案**：
```bash
# 1. 检查配置
cat .env

# 2. 测试连接
npm run test-connection

# 3. 查看日志
npm run logs
```

---

### 问题 2: 检索结果不准确

**可能原因**：
- 查询太模糊
- 记忆未正确分类
- 索引未更新

**解决方案**：
```bash
# 1. 使用更精确的查询
"昨天讨论的 Memory-Master 项目" vs "那个项目"

# 2. 重建索引
npm run rebuild-index

# 3. 手动分类记忆
编辑 MEMORY.md，移动到正确的分类
```

---

### 问题 3: 性能变慢

**可能原因**：
- 记忆太多
- 索引过大
- 未定期压缩

**解决方案**：
```bash
# 1. 运行压缩
npm run compact

# 2. 清理旧记忆
npm run cleanup --days=30

# 3. 检查性能
npm run performance-report
```

---

### 问题 4: 记忆丢失

**可能原因**：
- 文件被误删
- 存储路径变更
- 同步问题

**解决方案**：
```bash
# 1. 检查回收站
# 2. 从备份恢复
npm run restore --backup=2026-04-06

# 3. 查看历史版本
git log memory/
```
